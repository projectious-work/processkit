# Migration MIG-20260415T085311

From `v0.15.0` to `v0.16.0` — Add `primary_contact`, `clone_cap`,
`cap_escalation` to Role schema.

## Summary

Extends the Role schema with three optional fields to support team
contact assignment and per-role parallelism control:

- `primary_contact` (boolean, default `false`): True for the single
  owner-facing contact role per team.
- `clone_cap` (integer, default `5`): Per-role concurrency ceiling.
  PM must be `1`.
- `cap_escalation` (string, default `"owner"`): Who approves exceeding
  `clone_cap` (literal `"owner"` or actor-ref like `ACTOR-<id>`).

## Affected files

### Schema changes

- `context/schemas/role.yaml` — Add three new properties to Role spec
  with descriptions, defaults, and validation notes.

### Skill documentation

- `context/skills/processkit/role-management/SKILL.md` — Update version
  from 1.0.0 to 1.0.1, add new fields to field-list table, add
  explanatory section on team contact and clone management, document
  validation invariants.

### Data migration

Existing Role entities under `context/roles/` will be updated:

- All roles receive `clone_cap: 5` and `cap_escalation: "owner"` as
  defaults.
- `ROLE-project-manager` (if present) receives `primary_contact: true`
  and `clone_cap: 1`.
- All other roles receive `primary_contact: false` (or remain unset per
  existing entity style).

## Validation invariants (schema + MCP server)

- Exactly zero or one Role in any team sets `primary_contact: true`.
- If `primary_contact: true` and `clone_cap: 1`, the role is the PM
  equivalent.
- `cap_escalation` must be non-empty when `clone_cap > 0`.
