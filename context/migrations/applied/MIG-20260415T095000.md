# Migration MIG-20260415T095000

From `v0.15.0` to `v0.16.0` — Add `is_template`, `templated_from`
to Actor schema.

## Summary

Extends the Actor schema with two optional fields to support the
team-creator's templating workflow. Templates are the 8 canonical
seed Actors that define role bindings; clones are spawned per-task
and back-reference their template.

- `is_template` (boolean, default `false`): True for the canonical
  seed Actor that defines a role binding; clones derive from it.
  Enables index queries that separate "the 8 seed team members" from
  "clones spawned for a specific task."
- `templated_from` (Actor-ref, default `null`): Back-reference from
  a clone to its template Actor. Null for templates themselves and for
  one-off non-template Actors.

## Affected files

### Schema changes

- `context/schemas/actor.yaml` — Add two new optional properties to

### Skill documentation

### Data migration

Existing Actor entities under `context/actors/` will be updated:

- All 8 `ai-agent` type Actors currently in `context/actors/` (the
  team-creator seeds: ACTOR-assistant, ACTOR-developer,
  ACTOR-jr-architect, ACTOR-jr-developer, ACTOR-jr-researcher,
  ACTOR-pm-claude, ACTOR-sr-architect, ACTOR-sr-researcher) receive
  `is_template: true`. These are the canonical templates.
- Any existing `human` type Actors receive `is_template: false`.
- All Actors receive `templated_from: null` (clones will be created
  later with non-null values).

## Validation invariants (schema + MCP server if needed)

- If `is_template: true`, `templated_from` MUST be null (templates
  cannot have templates).
- If `templated_from` is non-null, it must reference an Actor that
  has `is_template: true` (clone must point at a real template).
- An Actor may be neither a template nor a clone — both fields retain
  their defaults.
