  title: aggregate-mcp lazy_catalog mode opts into shared GatewayRegistry via env
    var
  state: accepted
  decision: 'Add a lazy_catalog mode to the aggregate-mcp server by reusing the shared
    `processkit.gateway.GatewayRegistry` with `import_mode=''lazy-catalog''` and `register_gateway_tools()`.
    Activation is opt-in via env var `PROCESSKIT_MCP_LAZY=1` (also accepts `PROCESSKIT_MCP_MODE=lazy_catalog`).
    Default remains eager for backward compatibility. Catalog source is the existing
    `processkit-gateway/mcp/tool-catalog.json` (also resolvable via `PROCESSKIT_MCP_CATALOG_PATH`
    override). Cache invalidation is build-time: catalog regenerated by the existing
    `processkit-gateway catalog --write` command and shipped in repo; lazy mode does
    NOT regenerate at startup.'
  context: 'gh#31 asks for lazy per-skill module imports in the aggregate-mcp server
    with fast ListTools discovery. ThriftyWren (BACK-20260502_0857) already shipped
    lazy-catalog primitives in `processkit.gateway` and wired them into the gateway
    server. Two design questions: (1) opt-in surface, (2) catalog cache strategy.'
  rationale: Reusing GatewayRegistry avoids duplicate import logic, keeps tool naming/collision
    rules consistent across gateway and aggregate surfaces, and means the catalog
    is the single source of truth. Env var opt-in matches the gh#31 acceptance criteria
    verbatim and preserves eager-mode users. Build-time catalog generation keeps cold-start
    fast (no skill walk) and avoids race conditions; the existing `tool-catalog.json`
    is already committed and refreshed by gateway tooling.
  alternatives:
  - option: Separate entrypoint (server_lazy.py)
    rejected_because: duplicates wiring; harder to test that eager and lazy share
      registration shape
  - option: On-demand catalog probe at startup
    rejected_because: requires importing every server to build the manifest, defeating
      the point of lazy mode
  - option: Hash-based catalog invalidation
    rejected_because: premature; build-time generation is sufficient because catalog
      ships in repo with the skills
  consequences: aggregate-mcp now has eager (default) and lazy_catalog modes. Operators
    flip PROCESSKIT_MCP_LAZY=1 to defer skill imports until first tool call. Tool
    surface and naming stay identical between modes. If a skill ships without a corresponding
    catalog entry the lazy server will not expose it until catalog is regenerated
    — surfaced clearly in list_aggregate_tools as runtime.catalog_path.
