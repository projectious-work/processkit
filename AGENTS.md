# AGENTS.md ‚Äî processkit

This is the canonical entry point for any AI agent (or human) working on
this repository. Agent harnesses that auto-load a provider-specific file
(`CLAUDE.md`, `CODEX.md`, `.cursor/rules`, etc.) treat that file as a
**thin pointer** to this one. All authoritative agent instructions for
processkit live here.

**processkit** is the content layer for [aibox](https://github.com/projectious-work/aibox).
It ships **19 process primitives**, **106 skills** (multi-artifact
packages), process templates, six Python MCP servers, a shared Python
lib, the PROVENANCE.toml provenance map, and supporting scripts. aibox
installs the content into projects when consumers pin a processkit tag.

processkit itself is **standalone** ‚Äî it does not depend on aibox.
aibox depends on processkit, not the other way around. The fact that
this repo's *current development* uses aibox to manage its dev container
is incidental, not architectural.

**Current tag:** v0.4.0 (2026-04-07).

## Critical: src/ vs everything else

**Everything under `src/` is shipped content.** Treat it like source code.
It is what consumers install when they pin a processkit tag. It must be
**provider-neutral** (no Claude/Codex/Cursor references) and
**aibox-neutral** (consumable without aibox).

**Everything outside `src/` is about THIS repo:** its own dev environment,
its own project management, its own docs site, its own contributor
tooling. This is where aibox-specific notes (the 0.14.1 pin, the
devcontainer relationship) live.

## The 6 MCP servers (v0.3.0+)

Six servers ship under `src/skills/<name>/mcp/server.py` ‚Äî all uv-runnable
PEP 723 inline-deps Python scripts using the official MCP SDK over STDIO:

| Skill | Layer | Tools |
|---|---|---|
| `index-management` | 0 | `reindex`, `query_entities`, `get_entity`, `search_entities`, `query_events`, `list_errors`, `stats` |
| `id-management` | 0 | `generate_id`, `validate_id`, `list_used_ids`, `format_info` |
| `event-log` | 0 | `log_event`, `query_events`, `recent_events` |
| `workitem-management` | 2 | `create_workitem`, `transition_workitem`, `query_workitems`, `get_workitem`, `link_workitems` |
| `decision-record` | 2 | `record_decision`, `transition_decision`, `query_decisions`, `get_decision`, `supersede_decision`, `link_decision_to_workitem` |
| `binding-management` | 2 | `create_binding`, `end_binding`, `query_bindings`, `resolve_bindings_for` |

## Quick commands

```bash
# Smoke-test all 6 MCP servers (fastest feedback loop)
uv run scripts/smoke-test-servers.py

# Diff between two tags using PROVENANCE.toml at each tag
scripts/processkit-diff.sh --from v0.3.0 --to v0.4.0
scripts/processkit-diff.sh --from v0.3.0 --to v0.4.0 --format toml

# Regenerate PROVENANCE.toml against current git history (release time)
scripts/stamp-provenance.sh v0.X.Y

# Validate the PROVENANCE.toml is up to date (CI check, no write)
scripts/stamp-provenance.sh --check v0.X.Y
```

## Working in this repo

**Building processkit content:** edit `src/skills/`, `src/primitives/`,
`src/processes/`, `src/packages/`, `src/scaffolding/`. These will be
validated by `aibox lint` once that lands (Phase 4.4 on the aibox side,
in flight).

**Provider-neutral and aibox-neutral content under `src/`:** anything
shipped must work for a consumer who never installs aibox or who uses a
non-Claude agent harness. Reference processkit's own internals (MCP
servers, the index, the lib, primitives) freely; do not reference
`aibox.toml`, `aibox sync`, `CLAUDE.md`, or any provider-specific path.

**Releasing:** semver tags. The full release process is documented in
`context/HANDOVER.md` section 14. Always run `scripts/stamp-provenance.sh`
before tagging ‚Äî the diff script's correctness depends on it.

**Style ‚Äî line widths:** hard-wrap markdown prose, Python, and YAML at
80 columns. Exemptions: tables (rows may be wide; wrap only between
rows), fenced code blocks, URLs and link references, YAML frontmatter,
TOML files. The full rationale and survey is in BACKLOG.md (under the
v0.5.0 housekeeping section once it lands). An `.editorconfig` at the
repo root encodes the rule for editors that respect it.

**aibox version pinning:** this repo pins `aibox 0.14.1` in `aibox.toml`.
That version still ships embedded skills, so it can bootstrap processkit
without a chicken-and-egg problem. **Do not bump aibox here until** (a)
aibox ships processkit-consumption logic AND (b) processkit has a tag
that the new aibox knows about. See HANDOVER ¬ß2 for the rationale.

## aibox-sync perimeter (do not get confused by historical entanglement)

aibox 0.14.1 originally generated several files in this repo when this
repo was bootstrapped. Today, the legitimate scope of `aibox sync` is
**exactly** these gitignored devcontainer files:

- `.devcontainer/Dockerfile`
- `.devcontainer/docker-compose.yml`
- `.devcontainer/devcontainer.json`

**No other file in this repo is in scope for aibox sync.** AGENTS.md,
CLAUDE.md, README.md, src/, context/, scripts/ ‚Äî none of these are
touched by `aibox sync`. If aibox ever does touch one of them, that is
an aibox-side bug to fix on the aibox side, not a processkit-side
concern. Do not pre-block on "but what if aibox sync overwrites X" ‚Äî
the answer is "it does not, by definition; if it did, it's a bug".

## Out of scope (read HANDOVER ¬ß15 for the full list)

Do NOT add: RBAC enforcement, signed commits, workflow execution engine,
deterministic event logging, container infrastructure, a `processkit`
CLI, or REST/gRPC tool interfaces. processkit is consumed (via git
tags), it does not run as a command. If you find yourself wanting any
of these, you are likely in the wrong project.
