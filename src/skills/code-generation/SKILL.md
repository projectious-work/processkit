# Code Generation

Patterns for generating code programmatically — template engines, AST manipulation,
scaffolding tools, and build-step generation.

## When to Use

- Building or extending a code generator or scaffolding tool.
- Choosing between template engines (Jinja2, Handlebars, EJS).
- Deciding whether to generate code or write an abstraction.
- Adding code generation as a build step in CI/CD.
- Implementing macro systems or AST-based transformations.

## Instructions

### Generate vs Abstract

**Generate when:** output spans multiple languages, must be zero-dependency, or the
mapping from input to output is mechanical and repetitive.

**Abstract when:** a shared library with configuration solves the problem, runtime
flexibility matters, or maintenance of generated files would exceed library maintenance.

### Template Engines

| Engine       | Language | Best for                                   |
|--------------|----------|--------------------------------------------|
| Jinja2       | Python   | Config files, IaC, multi-language output    |
| Handlebars   | JS/TS    | Logic-less templates, client-side rendering |
| EJS          | JS/TS    | Embedded JS logic, quick prototypes         |
| Tera         | Rust     | Rust-native projects, similar to Jinja2     |
| Go templates | Go       | Go projects, kubectl/Helm templates         |

- Keep templates logic-free — push logic into the data model.
- Use template inheritance/blocks for shared structure across variants.
- Store templates alongside the generator, not in the output directory.
- Emit a header comment: `// Code generated by <tool> — DO NOT EDIT.`

### AST Manipulation

Use language-specific parsers (`syn` for Rust, `ast` for Python, `ts-morph` for TS)
when transformations depend on syntactic context, must guarantee valid output, or are
too complex for regex. Parse into AST, transform the tree, emit back to source.

### Macro Systems

Rust `proc_macro`, Lisp macros, C preprocessor:
- Prefer declarative macros over procedural when the pattern is simple.
- Document macro expansion — provide `cargo expand` or equivalent for inspection.
- Keep generated code minimal; delegate to library functions for complex logic.

### Scaffolding Tools

For cookiecutter, yeoman, `cargo-generate`:
- Define templates with placeholder variables, sensible defaults, and validation.
- Scaffolded code must compile and run immediately — include tests, linting, CI config.

### Build-Step Generation

- Store the source-of-truth (schema, spec, DSL) in version control.
- Fail the build if generated output differs from committed files.
- Pin the generator version. Cache intermediate artifacts for speed.

### Keeping Generated Code in Sync

- Mark generated files: header comment, `.generated.` in filename, or a manifest.
- Single command to regenerate: `make generate` / `just generate`.
- CI check that runs generation and diffs against committed files.
- Never hand-edit generated files — fix the template or input instead.

## Examples

### Example 1: Generate API clients from an OpenAPI spec

```
User: We have an OpenAPI spec and need TypeScript and Python clients.

Agent: Reads the spec, evaluates options (openapi-generator vs openapi-typescript).
  Chooses openapi-generator for multi-language support. Configures generate.sh
  producing both clients with "DO NOT EDIT" headers and a CI step that fails
  if output differs from committed files.
```

### Example 2: Decide between codegen and a library

```
User: We repeat the same 40-line DB access pattern in every service.
      Should we generate it or write a shared library?

Agent: All services use Python; variation is just table name and columns.
  Recommends a shared library: single language eliminates codegen advantage,
  library allows runtime changes, one library < N generated files to maintain.
```

### Example 3: AST-based migration of deprecated API calls

```
User: Rename log_event() to emit_event() across 200 files. Find-replace
      hits false positives in strings and comments.

Agent: Writes a ts-morph codemod that parses each file, finds call
  expressions where the callee is log_event (skipping strings/comments),
  renames to emit_event, and writes back preserving formatting.
```
