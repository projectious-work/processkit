# actor-profile MCP server

Manages Actor entities — humans, AI agents, and services that
participate in the project.

## Tools

| Tool | Purpose |
|---|---|
| `create_actor(name, type, email?, handle?, expertise?, roles?, preferences?, joined_at?)` | Create a new Actor entity |
| `get_actor(id)` | Fetch a single Actor by ID |
| `update_actor(id, **fields)` | Update fields on an existing Actor |
| `deactivate_actor(id, left_at?)` | Mark an Actor inactive (sets active=false + left_at) |
| `list_actors(type?, active_only?, expertise?, limit?)` | List Actors with optional filters |

`type` must be one of `human`, `ai-agent`, `service`. The MCP server
validates the spec against `src/primitives/schemas/actor.yaml` before
writing.

## Storage

Actor entities live at `<project-root>/context/actors/<id>.md`. The ID
is generated by the id-management module using the project's configured
ID format.

## Running

```bash
uv run context/skills/actor-profile/mcp/server.py
```

## Notes

Actors are **descriptive, not restrictive** — there is no RBAC
enforcement attached to roles or expertise (DEC-017). For scoped or
time-bounded role assignments, use a Binding via binding-management
rather than the unscoped `roles` field on the Actor.

Deactivation, not deletion: when an actor leaves, call
`deactivate_actor` rather than removing the file. Historical events
and bindings reference the actor file.
